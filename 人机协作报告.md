#### 1. 构建监督学习数据集（聚合 + 8:2 划分）.

- **核心Prompt**：“根据我们前面处理的相关数据，帮我构建预测某区域某时段出行需求量的数据集，并把训练集和测试集按8:2分开”

- **决策**：采纳！运行后得到聚合样本数 22058，训练集 17646，测试集 4412，比例正好 8:2，没有问题！



#### 2. 神经网络模型构建与训练

- **核心Prompt**：“请用 TensorFlow 构建神经网络，训练并画 loss 曲线，报告 MAE 与 RMSE 的情况”

- **AI 输出摘要**：给出 3 层全连接网络（64→32→16→1，ReLU 激活），用 `StandardScaler` 对特征做标准化，编译参数 `optimizer='adam', loss='mse'`，训练 50 轮，batch size 128，记录训练/验证 loss，最后用 `matplotlib` 绘制 loss 曲线并保存，并在测试集上计算 MAE、RMSE

- **决策**：采纳！运行过程中虽然出现多条 TensorFlow 的 WARNING（oneDNN、GPU 不可用等），但是给DS分析后，DS解释这些仅表示 Windows 下用的是 CPU 且有些优化选项开启，不会影响结果

- **错误与修正**：通过观察训练结果我发现到验证集 loss 略低于训练集 loss（应该是有轻微异常），DS 解释可能是 dropout 缺失或验证集规模小造成的统计波动，并未出现过拟合，所以应该是没有什么问题的



#### 3. Loss 曲线与模型评估

- **核心Prompt**：“（结果截图）这个是结果的截图，loss 曲线的情况如何？MAE 和 RMSE 分别是多少？”

- **AI 输出摘要**：曲线图已自动生成并保存到 `outputs/nn_loss_curve.png`，训练 loss 和验证 loss 均随 epoch 下降并趋平，模型收敛正常；最终测试集结果为：`MAE = 163.18，RMSE = 256.00`。DS 分别向我解释了 MAE（平均预测偏差 163 单）和 RMSE（对大误差敏感的指标 256 单）的含义

- **决策**：将这两个数值记录下来用于后面与随机森林对比



#### 4. 随机森林对比实验

- **核心Prompt**：“请与随机森林进行对比实验，请帮我加上随机森林训练和评估的代码，并分析这两种方法在我们现在处理的任务中的优劣”

- **AI 输出摘要**：添加了 `RandomForestRegressor(n_estimators=100, random_state=42)`，预测测试集并计算 MAE、RMSE，然后提供了一个清晰的对比汇总表，并自动输出结论：**随机森林 RMSE 更低（49.56 < 256.00），在此任务上表现更优**

- **决策**：采纳！随机森林 MAE 仅 18.62，RMSE 49.56，远优于神经网络


- **AI 帮助分析优劣**：  
  DeepSeek 从三个方面解释了随机森林更优的原因：

  3. **非线性程度**：任务规律相对简单，随机森林的“分段投票”比神经网络全局拟合更直接有效。

#### 2. 5 种问题类型的规则匹配

- **核心Prompt**：“需要支持时段查询、区域排名、需求预测、费用查询、速度/拥堵查询 5 种类型，应该可以通过关键词匹配来实现，请完成这部分代码”

- **AI 输出摘要**：为每种类型设计了关键词组合（如“小时+最多”匹配时段查询，“区域+热门”匹配区域排名），用 `if any(...) in q` 判断用户意图，返回对应答案。答案数据直接从 M1-M3 的变量中提取，如 `df['pickup_hour'].value_counts().idxmax()` 获取订单最多的小时，`y_pred_rf.mean()` 返回随机森林预测的平均需求量，图表路径也一并返回

- **决策**：采纳！测试中发现“预测需求”问题时，有时用户输入较模糊（如只说“预测”），可能被误匹配到其他规则，所以在与 DS 讨论后加入了下面的增强功能



#### 3. 增强功能：用户可主动调用大模型

- **核心Prompt**：“我发现，有时候我希望能用大模型回答，但系统走了规则，感觉回答起来差点意思，能不能加个功能，让我说‘帮我分析’时强制走大模型？”

- **AI 输出摘要**：DS 建议在 `rule_based_answer` 函数最前面增加判断：如果问题中包含“帮我分析”“详细解释”等关键词，直接返回 `(False, "", None)`，跳过规则匹配，强制进入大模型兜底逻辑

- **决策**：采纳！初次实现时因缩进错误导致运行失败，AI 指出 `return` 语句需比 `if` 多一级缩进，修正后功能正常可以运行，这个改进让用户可以自主选择走规则还是走大模型，提升了系统的灵活性

#### 5. System Prompt 的设计与迭代过程

本次 System Prompt 经历了三版迭代：

**v1（初始版本）**：你是一个数据分析助手，回答关于纽约出租车数据的问题

**测试效果**：回答泛泛而谈，没有具体的结合我们的程序和数据进行回答，专业性差点意思
**发现问题**：应该是因为我们没有给大模型限定数据范围，模型不知道要基于具体数据集回答，所以回答不够专业也不够具体


**v2（增加数据约束）**：你是一个纽约出租车数据分析助手，基于2023年1月黄色出租车数据（约293万条行程）回答问题，回答时请简洁、数据驱动

**测试效果**：回答更有针对性，但遇到一些奇怪问题时模型会编造数据  
**发现问题**：没有设置“能力边界”，模型不懂说“不知道”，应该将模型的角色设置要求的具体一些

## 二、三阶段对比——以「提取小时特征」为例

我选取了 M1 特征提取中的"从上车时间中提取小时"这一个相对简单的功能，来对比三种开发模式的差异


### Native 版
```python
# Native 版：提取小时特征
data = []
# 用with open 函数打开数据文件，逗号分隔
with open("data/yellow_tripdata_2023-01.csv", "r") as f:
    header = f.readline().strip().split(",")
    for line in f:
        row = line.strip().split(",")
        data.append(row)

# 找到 pickup_datetime 的索引（用for循环）
pickup_idx = -1
for i, col in enumerate(header):
    if col == "tpep_pickup_datetime":
        pickup_idx = i
        break

# 提取小时
hours = []
for row in data:
    datetime_str = row[pickup_idx]
    # 格式类似 "2023-01-01 00:32:10"
    time_part = datetime_str.split(" ")[1]
    hour = time_part.split(":")[0]
    hours.append(int(hour))

print("前 10 个行程的小时：", hours[:10])
# 最后输出
```

### Prompt 版
**提示词**："请帮我写一段 Python 代码，用 pandas 读取 data/yellow_tripdata_2023-01.parquet 文件，该文件我放在目录下的data文件夹内，从数据文件的 tpep_pickup_datetime 列提取小时，存为新列 pickup_hour，然后打印前 10 行的小时值"（写提示词的时候有参考完整的代码）

```python
# Prompt 版：提取小时特征
import pandas as pd

df = pd.read_parquet("data/yellow_tripdata_2023-01.parquet")
df['pickup_hour'] = df['tpep_pickup_datetime'].dt.hour
print(df['pickup_hour'].head(10))
```
### Vibe 版
**Prompt**：“我想从行程时间中提取小时、星期、是否高峰等特征，应该如何实现？”

```python
# Vibe 版：提取多个时间特征
df['pickup_hour'] = df['tpep_pickup_datetime'].dt.hour
df['pickup_weekday'] = df['tpep_pickup_datetime'].dt.weekday
df['is_peak'] = df['pickup_hour'].isin([7, 8, 9, 17, 18, 19]).astype(int)
print(df[['pickup_hour', 'pickup_weekday', 'is_peak']].head())
```

### 对比分析：

这三个版本的代码有不少区别
- **效率**：Native版本的效率应该是最低的，毕竟是纯人工手写，肯定需要耗费一定的时间和精力；相比之下Prompt版和Vibe版效率要高很多，只需要给出相应的要求，大模型会自己写好代码，对于编程的人来说可以节省一定的精力，但是在大模型给出代码之后依旧需要自己进行人工审查和理解代码的运行逻辑

- **理解深度**：写Native版时，程序员对代码的理解深度应该是最深最充分的，只有充分理解需求，充分理解每行代码背后的运行逻辑，才能自己手工写出代码。Prompt版也需要了解很多语法的运行逻辑和需求，给出明确的指令，才能精确地写出提示词，让AI精确地生成相应的代码。相比之下，Vibe版更加友好，即使对语法和逻辑了解的没有那么全面和透彻，也可以通过精确的描述任务需求，来给AI下达指令，最后也能做出相当的成功（不过这依旧要求使用者对AI生成的代码有一定的审查能力！），所以综合来说，Vibe版对程序员的要求没有前两个那么高。
